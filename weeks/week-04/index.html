MON

Capstone Planning

1.5 hours

Why this matters: A successful capstone starts with a clear problem statement, dataset selection, and evaluation plan. This day turns ideas into a reproducible project blueprint.

📓 Day1_Capstone_Planning.ipynb

🎯 Problem framing

Define inputs, target, and success metric first. If the goal is house price prediction, the project steps differ from a classification capstone. Keep the scope narrow and measurable.

🧠 Dataset sanity check

Inspect row counts, feature types, missingness, and label balance. Use a small sample first and verify the dataset is suitable before investing time in modeling.

🛠️ Project structure

Create folders for data/, notebooks/, src/, and outputs/. Document the pipeline steps so you can reproduce the capstone later.

📊 Evaluation plan

Choose metrics that match the problem: RMSE for regression, F1 for class imbalance, AUC for ranking. Build the metric into the notebook from day one.

KEY TAKEAWAYS

  • Write the project question in one sentence: what you predict and why it matters.
  • Dataset quality is more important than model complexity.
  • Document the train/test split strategy before doing any feature engineering.
  • Establish a validation metric that aligns with the business goal.
  • Capstone planning reduces rework and keeps the week focused.
TUE

Data Preprocessing Review

1.5 hours

Why this matters: This day refreshes the core feature engineering patterns that turn raw data into model-ready input: missing data, encoding, scaling, and target-aware transformation.

📓 Day2_Data_Preprocessing_Review.ipynb

🧩 Missing values

Choose between dropna(), imputation, and indicator flags. Always compute missingness before splitting and avoid leaking future values into train statistics.

🔢 Scaling

Scale numeric features after splitting. Use StandardScaler for regression and most tree-based models, RobustScaler for heavy outliers.

🔠 Encoding

Use one-hot encoding for low-cardinality categoricals and ordinal mapping for natural order. For many categories, prefer target encoding or frequency encoding inside a proper CV loop.

🔄 Feature review

Revisit engineered features from Week 3, then prioritize those with the strongest correlation to your target. Keep the pipeline simple and explainable.

KEY TAKEAWAYS

  • Fit preprocessors only on training data, then apply to validation/test sets.
  • Use pd.Categorical or OneHotEncoder(drop='first') for categorical features.
  • Keep feature names consistent across train and inference pipelines.
  • Save imputer values, encoder mappings, and scaler parameters with the model pipeline.
  • Preprocessing is the most common source of pipeline bugs — document every transformation.
WED

Modeling & Pipeline Design

1.5 hours

Why this matters: An end-to-end capstone is only as strong as its reproducible pipeline. Today you design the model flow, select baseline algorithms, and build the first training pipeline.

📓 Day3_Modeling_and_Pipeline_Design.ipynb

🔧 Baseline first

Start with a simple model like LinearRegression or LogisticRegression. Baseline performance tells you whether your features are strong enough before you add complexity.

🧵 Pipeline patterns

Use Pipeline and ColumnTransformer to combine preprocessing + model training. This is the foundation for deployment-ready ML code.

📈 Model comparison

Compare models by the same metric on the same validation split. Avoid comparing apples and oranges by using a consistent preprocessing pipeline.

🗂️ Reproducibility

Set random seeds in preprocessors and model constructors. Save the exact pipeline configuration so you can reproduce training later.

KEY TAKEAWAYS

  • A complete pipeline includes preprocessing, feature transformation, and a model in one object.
  • Baseline algorithms are your sanity check — complex models should outperform them clearly.
  • Keep validation splits separate from the final test set.
  • Use named transformers to make hyperparameter search easier.
  • Document which model is selected and why before moving to deployment prep.
THU

Evaluation & Cross-Validation

1.5 hours

Why this matters: Evaluation turns a promising model into a reliable one. Today you measure performance, detect overfitting, and make your final model selection with cross-validation.

📓 Day4_Evaluation_and_CV.ipynb

📏 Metrics matter

Select metrics that reflect the business goal. Use RMSE for regression and F1/ROC AUC for classification. One metric avoids ambiguity.

🔁 Cross-validation

Use cross_val_score with your full pipeline to measure real generalization. This is the safest way to compare preprocessing + model combinations.

🧪 Validation curves

Plot training vs validation performance across complexity. If the gap grows, the model is overfitting; if both scores are low, the model underfits.

✅ Final holdout

Reserve a clean holdout set for the final score. Do not use it for hyperparameter tuning or model selection.

KEY TAKEAWAYS

  • Cross-validation must include preprocessing inside the fold, not applied beforehand.
  • Choose a primary metric and use secondary metrics for additional insight.
  • Use stratified splits for imbalanced classification datasets.
  • Final model selection should happen before evaluating on holdout data.
  • Save the evaluation code so you can reproduce the final reported score.

SAT

Deployment Prep

1.5 hours

Why this matters: A completed ML project is deployable. Today you package your pipeline, export the model, and prepare prediction code for a production-like workflow.

📓 Day6_Deployment_Prep_and_Pipelines.ipynb

📦 Save the pipeline

Use joblib.dump() to store the entire trained pipeline. This includes preprocessing, feature transformers, and the final estimator in one reusable object.

🔄 Inference code

Write a minimal function that accepts raw data, applies the saved pipeline, and returns predictions. Production code should not repeat preprocessing logic manually.

🧯 Input checks

Validate feature columns, dtypes, and missing values before prediction. A small guard clause prevents bad input from breaking the pipeline.

🌐 Deployment story

Map your pipeline to a simple deployment target: an API, scheduled batch job, or Azure ML endpoint. The notebook should make it easy to understand the next step.

KEY TAKEAWAYS

  • Save the entire pipeline, not just the model weights.
  • Package preprocessing + model into a single object for inference.
  • Write clear input validation around the prediction entry point.
  • Deployment readiness is the final test of a capstone project.
  • Keep deployment notes inside the notebook for easy handoff.

Week 4 Flashcards

20 rapid knowledge checks for capstone planning, preprocessing, modeling, evaluation, and deployment.

Open cards: 0/20
1 Planning What is the first question you should answer when starting a capstone project? ▼
Define the problem clearly: what is being predicted, what data is available, and which metric will determine success.
2 Preprocessing Why do you set a fixed random seed in a reproducible pipeline? ▼
A fixed seed ensures that data splits, model initialization, and stochastic operations produce the same results each time, making experiments repeatable.
3 Preprocessing What is the safest way to handle missing values in a modeling pipeline? ▼
Impute missing values using a transformer inside the pipeline, fitted on training data only, and then apply the same transform to validation/test data.
4 Modeling What does a scikit-learn Pipeline combine? ▼
A Pipeline combines preprocessing steps and an estimator into a single object so you can call fit, predict, and score with one unified interface.
5 Modeling Why use ColumnTransformer in an ML pipeline? ▼
ColumnTransformer lets you apply different preprocessing steps to numeric and categorical columns in the same pipeline.
6 Evaluation What is the role of a holdout dataset? ▼
The holdout dataset provides a final unbiased estimate of model performance after training and validation are complete.
7 Evaluation Why is cross-validation better than a single train/test split? ▼
Cross-validation averages performance over multiple splits, reducing variance and giving a more stable estimate of model generalization.
8 Planning What should be included in your capstone README? ▼
The README should explain the problem, data source, evaluation metric, pipeline steps, model choice, and how to reproduce the results.
9 Deployment What is the key advantage of saving the full fitted pipeline? ▼
Saving the full fitted pipeline preserves preprocessing logic and model parameters together, avoiding mismatched training and inference transforms.
10 Deployment Why should inference code validate input columns? ▼
Validating input columns prevents silent failures or incorrect predictions when the feature set changes in production.
11 Planning What is the simplest way to decide if a dataset is fit for a regression capstone? ▼
Check if the target is continuous, has sufficient variance, and whether the features have predictive signal for the target.
12 Preprocessing When should you use RobustScaler instead of StandardScaler? ▼
Use RobustScaler when numeric features contain outliers, because it scales based on the median and interquartile range.
13 Modeling What is the primary benefit of starting with a linear baseline model? ▼
Linear baselines are fast, interpretable, and set a reference point for whether more complex models are actually improving performance.
14 Evaluation What does stratified splitting preserve? ▼
Stratified splitting preserves the class distribution between train and test sets for classification problems.
15 Deployment What file format is commonly used to save scikit-learn pipelines? ▼
Joblib is commonly used to serialize scikit-learn pipelines with joblib.dump() and joblib.load().
16 Capstone What is the final check before declaring the capstone complete? ▼
Confirm that the pipeline is reproducible, the final model is evaluated on holdout data, and the inference code works end-to-end.
17 Preprocessing Why should you keep categorical feature names stable? ▼
Stable feature names prevent mismatches between training and inference, especially when using one-hot encoding or feature selection.
18 Modeling What is a good sign that your capstone model is overfitting? ▼
The training score is much higher than the validation score, and validation performance worsens as the model complexity increases.
19 Evaluation What should you do if the chosen metric is unstable across folds? ▼
Investigate data leakage, ensure consistent preprocessing across folds, and consider whether the metric or features are too sensitive to small sample changes.
20 Deployment What does a production-ready inference wrapper need to do? ▼
It should validate raw input, load the saved pipeline, apply preprocessing and prediction, and return results in a consistent format.
