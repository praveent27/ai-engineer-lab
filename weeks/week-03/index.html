Handling Missing Data

1.5 hours

Why this matters: Real-world datasets always have missing values. How you handle them determines whether your model learns signal or noise. Deleting rows throws away information. Imputing wrong introduces bias. Getting this step right is often the single largest leverage point in a feature engineering pipeline.

📓 Day1_Handling_Missing_Data.ipynb

🔍 MCAR / MAR / MNAR — The Three Types of Missingness

MCAR (Missing Completely At Random): probability of missing is unrelated to any value — e.g., random sensor dropout. Safe to delete rows or do simple imputation. MAR (Missing At Random): probability depends on observed data — e.g., older patients skip a survey question. Impute using other features. MNAR (Missing Not At Random): probability depends on the missing value itself — e.g., high-income earners skip the income field. Hardest to handle; requires domain knowledge or multiple imputation.

📊 Detecting and Profiling Missingness

df.isnull().sum() counts per-column. df.isnull().mean() * 100 gives percentages. msno.matrix(df) (missingno library) visualises patterns across rows and columns — look for correlated missingness (two columns always missing together suggests MNAR). Always build a missing % report before imputing anything.

🩹 Simple Imputation Strategies

Numeric — mean: use when distribution is approximately normal and few values are missing. Numeric — median: prefer for skewed distributions or when outliers are present (median is robust). Categorical — mode: fill with the most frequent category. Forward/Backward fill: for time-series data where the previous value is a good estimate. Use SimpleImputer(strategy='median') in scikit-learn.

🤖 Advanced: KNN & Iterative Imputation

KNNImputer(n_neighbors=5) fills each missing value using the mean of k nearest complete neighbours. Captures feature correlations. IterativeImputer (MICE — Multiple Imputation by Chained Equations) models each feature with missing values as a function of all others — iteratively refines estimates. More accurate but slower. Use for datasets where missingness is moderate (5–40% of values).

🚩 Missing Indicator Features

Adding a binary column feature_was_missing (1 if original was NaN, 0 otherwise) lets the model learn that missingness itself carries signal. When MNAR or MAR, the model can pick up that absence of a value is predictive. Always pair with imputation — don't just add indicators. In scikit-learn: SimpleImputer(add_indicator=True).

IMPUTATION STRATEGY DECISION GUIDE

Numeric, normal dist, <5% missing→Mean imputation
Numeric, skewed / outliers→Median imputation
Categorical→Mode imputation
Time series→Forward fill (ffill)
Features correlated, moderate missing→KNNImputer
Complex dependencies, <40% missing→IterativeImputer (MICE)
Missingness carries signal (MNAR)→add_indicator=True
from sklearn.impute import SimpleImputer, KNNImputer, IterativeImputer
import pandas as pd
import numpy as np

# Profile missingness
missing_report = pd.DataFrame({
    'count': df.isnull().sum(),
    'pct': df.isnull().mean() * 100
}).sort_values('pct', ascending=False)

# Simple imputation
num_imputer = SimpleImputer(strategy='median', add_indicator=True)
cat_imputer = SimpleImputer(strategy='most_frequent')

# KNN imputation (captures correlations)
knn_imp = KNNImputer(n_neighbors=5)

# MICE — iterative imputation
mice_imp = IterativeImputer(max_iter=10, random_state=42)

KEY TAKEAWAYS

  • Always understand why data is missing before deciding how to handle it — strategy depends on missingness mechanism (MCAR/MAR/MNAR).
  • Use median over mean for numeric imputation unless you've verified the feature is symmetric — median is robust to outliers.
  • Never impute on the full dataset before train/test split — you'll leak test statistics into training. Fit imputers on training data only, then transform both.
  • Missing indicator columns are free information — add them whenever missingness could be predictive.
  • Dropping rows is rarely a good strategy — unless the feature is completely missing for that row and cannot be imputed.
TUE

Categorical Encoding

1.5 hours

Why this matters: Most real datasets have categorical columns — city names, product categories, gender. ML models only understand numbers. The encoding strategy you choose dramatically changes what the model learns. Wrong encoding can create false ordinal relationships, explode dimensionality, or cause target leakage.

📓 Day2_Categorical_Encoding.ipynb

🔢 Ordinal Encoding — When Order Exists

Use when categories have a natural order: cold < warm < hot, or low < medium < high. OrdinalEncoder(categories=[['low','medium','high']]) maps to integers 0, 1, 2. Never use this for nominal categories (like city names) — it implies an ordering that doesn't exist and will mislead the model.

🔥 One-Hot Encoding — Nominal, Low Cardinality

Creates one binary column per category: no ordinal assumption. Best for nominal features with few unique values (<15 categories). OneHotEncoder(drop='first', sparse_output=False) — dropping one column prevents multicollinearity (the dummy variable trap). Watch out for cardinality explosion: a 1000-category feature creates 999 columns.

📉 Target Encoding — High Cardinality

Replace each category with the mean of the target variable within that category. E.g., city "London" → mean house price in London. Extremely powerful for tree models. Risk: target leakage. Always use K-fold cross-encoding (encode training fold using out-of-fold statistics). category_encoders.TargetEncoder handles this. Never fit target encoder on full training data.

🧮 Frequency / Count Encoding

Replace each category with how often it appears in the dataset. Useful for high-cardinality features without the leakage risk of target encoding. df['city_freq'] = df['city'].map(df['city'].value_counts()). Preserves signal about rare vs common categories. Works well with tree-based models.

🏷️ Binary & Hash Encoding

Binary encoding: converts integer ordinal code to binary digits — fewer columns than OHE. Useful for medium cardinality (15–100 categories). Hash encoding: uses a hash function to map categories to a fixed number of columns. Handles new categories at inference time (OHE can't). Risk of hash collisions. Both from category_encoders library.

ENCODING DECISION MATRIX

Has natural order (size, grade)→OrdinalEncoder
Nominal, <15 categories→OneHotEncoder(drop='first')
Nominal, 15–100 categories→BinaryEncoder
High cardinality (>100) + regression/trees→TargetEncoder (cross-fold)
High cardinality, avoid leakage→FrequencyEncoder
Unknown categories at inference→HashingEncoder
from sklearn.preprocessing import OrdinalEncoder, OneHotEncoder
import category_encoders as ce

# Ordinal: explicitly define category order
oe = OrdinalEncoder(categories=[['low', 'medium', 'high']])
X[['size_enc']] = oe.fit_transform(X[['size']])

# One-Hot: nominal, low cardinality
ohe = OneHotEncoder(drop='first', sparse_output=False, handle_unknown='ignore')
X_ohe = ohe.fit_transform(X[['colour']])

# Target encoding (cross-validated to avoid leakage)
te = ce.TargetEncoder(cols=['city'], smoothing=1.0)
X['city_enc'] = te.fit_transform(X['city'], y)

# Frequency encoding (no leakage)
freq_map = X_train['city'].value_counts().to_dict()
X['city_freq'] = X['city'].map(freq_map).fillna(0)

Common Trap: The Dummy Variable Trap

One-hot encoding 3 categories creates 3 columns that sum to 1 — perfect multicollinearity. Always use drop='first' or drop='if_binary' with linear models. Tree-based models are immune (they don't rely on matrix inversion), but dropping first is still good practice.

KEY TAKEAWAYS

  • Ordinal encoding for ordered categories, OHE for nominal low-cardinality — never swap these.
  • Target encoding is powerful but leaks if not done with cross-validation — always use K-fold encoding in production.
  • For high-cardinality features (>100 unique values), OHE creates too many columns — use frequency or target encoding.
  • Always handle unknown categories: handle_unknown='ignore' in OHE, else inference on new categories will crash.
  • Fit encoders only on training data. Never fit on the full dataset.
WED

Feature Scaling & Normalisation

1.5 hours

Why this matters: Many algorithms (linear models, SVMs, KNN, neural networks, PCA) are sensitive to feature magnitude. A salary column in thousands next to an age column in tens creates massive gradient imbalances. Scaling ensures all features contribute equally to distance calculations and gradient updates.

📓 Day3_Feature_Scaling.ipynb

📏 StandardScaler (Z-score Normalisation)

z = (x - μ) / σ — transforms data to have mean=0, std=1. Values typically fall in [-3, 3]. Best for: linear regression, logistic regression, SVMs, neural networks, PCA. Assumes features are approximately Gaussian. Not robust to outliers — a single extreme value shifts the mean and inflates std.

📐 MinMaxScaler (Min-Max Normalisation)

x_scaled = (x - min) / (max - min) — maps values to [0, 1] (or custom range). Use when you need bounded output (e.g., for neural networks with sigmoid/tanh activation, or image pixel values). Very sensitive to outliers — one extreme value compresses everything else to near zero.

💪 RobustScaler — Outlier-Resistant

x_scaled = (x - median) / IQR — uses median and interquartile range instead of mean and std. Robust to outliers — extreme values don't distort the scale. Ideal for features with heavy tails or known outliers (income, property prices, transaction amounts). Output is not bounded but typically in [-3, 3] for most values.

📊 Normalizer — Row-Wise Scaling

Unlike others (which scale columns), Normalizer scales each row to unit norm. Each sample vector has L2-norm = 1. Use when the magnitude of an individual sample doesn't matter but the direction does — text TF-IDF vectors, user behaviour vectors for cosine similarity search.

📈 Power Transforms — Fixing Skewed Distributions

PowerTransformer(method='yeo-johnson') applies a non-linear transformation to make distributions more Gaussian. The Yeo-Johnson method works for both positive and negative values. Box-Cox requires strictly positive values. Use before StandardScaler on skewed features (income, prices, counts). QuantileTransformer(output_distribution='normal') is an alternative that maps to normal by rank.

SCALE-SENSITIVE ALGORITHMS
  • Linear / Logistic Regression
  • Support Vector Machines (SVM)
  • K-Nearest Neighbours (KNN)
  • Principal Component Analysis (PCA)
  • Neural Networks
  • Gradient descent in general
  • K-Means Clustering
  • Ridge / Lasso Regression
SCALE-INVARIANT ALGORITHMS
  • Decision Trees
  • Random Forest
  • Gradient Boosting (XGBoost, LightGBM)
  • Naive Bayes
  • Tree-based models in general
from sklearn.preprocessing import StandardScaler, MinMaxScaler, RobustScaler
from sklearn.preprocessing import PowerTransformer, QuantileTransformer

# Standard: for Gaussian-ish features, linear/SVM/NN
scaler = StandardScaler()
X_scaled = scaler.fit_transform(X_train)

# MinMax: for bounded output [0,1]
mm_scaler = MinMaxScaler(feature_range=(0, 1))

# Robust: when you have outliers
rb_scaler = RobustScaler()  # uses median + IQR

# Fix skewness before scaling
pt = PowerTransformer(method='yeo-johnson')
X_transformed = pt.fit_transform(X[['income', 'price']])

# ALWAYS fit on train, transform train AND test
X_train_scaled = scaler.fit_transform(X_train)
X_test_scaled = scaler.transform(X_test)  # NOT fit_transform!

KEY TAKEAWAYS

  • Tree-based models (Random Forest, XGBoost) don't need scaling — they split on thresholds, not magnitudes.
  • Never fit_transform on test data — always fit on training data and only transform test data.
  • StandardScaler first, RobustScaler when outliers are present, MinMaxScaler when you need [0,1] bounds.
  • Apply PowerTransformer to fix skewed distributions before StandardScaler for best results with linear models.
  • Check skewness with df.skew() — values beyond ±1 are moderately skewed, beyond ±2 severely skewed.
THU

Feature Selection

1.5 hours

Why this matters: More features is not always better. Irrelevant features add noise, slow training, cause overfitting, and make models harder to interpret. The curse of dimensionality means that in high-dimensional spaces, distances become meaningless and models struggle. Selecting the right features is often more impactful than choosing the right algorithm.

📓 Day4_Feature_Selection.ipynb

📊 Filter Methods — Statistics-Based

Evaluate each feature independently of the model. Fast and cheap. Variance threshold: VarianceThreshold(threshold=0.01) removes near-constant features (they carry no signal). Correlation filter: remove one of any pair with |correlation| > 0.95 (they're redundant). Univariate tests: SelectKBest(f_classif, k=20) for classification, SelectKBest(f_regression, k=20) for regression. Use chi-squared for categorical features vs. categorical target.

🤖 Wrapper Methods — Model-Based Search

Use a model's performance to evaluate feature subsets. RFE (Recursive Feature Elimination): RFE(estimator=LogisticRegression(), n_features_to_select=10) — iteratively removes the least important feature. Computationally expensive but accurate. RFECV: same but cross-validated — automatically finds the optimal number of features. Use for datasets with <1000 features.

🌲 Embedded Methods — Feature Importance

Feature selection built into the model during training. L1 (Lasso) regularisation drives unimportant weights to zero — features with zero coefficients are eliminated. Tree feature importance: model.feature_importances_ in Random Forest or XGBoost measures how much each feature reduces impurity. SelectFromModel(threshold='mean') keeps features above threshold. Most efficient approach for tree-based models.

🎯 Mutual Information

mutual_info_classif / mutual_info_regression measures non-linear dependency between feature and target (unlike correlation which only captures linear). Returns I(X; Y) — how much knowing X reduces uncertainty about Y. Handles both linear and non-linear relationships. Use via SelectKBest(mutual_info_classif, k=20).

🔗 Removing Correlated Features

Highly correlated features add redundancy without new information — they can destabilise linear model coefficients. Compute correlation matrix: df.corr().abs(). Find upper triangle pairs above threshold. Drop one from each high-correlation pair. For feature groups, use domain knowledge to decide which to keep. In tree models correlation is less harmful but still reduces interpretability.

from sklearn.feature_selection import (
    VarianceThreshold, SelectKBest, f_classif,
    mutual_info_classif, RFE, SelectFromModel, RFECV
)
from sklearn.ensemble import RandomForestClassifier
import numpy as np

# 1. Remove near-constant features
sel = VarianceThreshold(threshold=0.01)
X_var = sel.fit_transform(X)

# 2. Remove highly correlated features
corr = pd.DataFrame(X).corr().abs()
upper = corr.where(np.triu(np.ones(corr.shape), k=1).astype(bool))
to_drop = [col for col in upper.columns if any(upper[col] > 0.95)]
X_uncorr = pd.DataFrame(X).drop(columns=to_drop)

# 3. Univariate selection
selector = SelectKBest(mutual_info_classif, k=20)
X_best = selector.fit_transform(X_train, y_train)

# 4. Tree feature importance
rf = RandomForestClassifier(n_estimators=100, random_state=42)
rf.fit(X_train, y_train)
feat_sel = SelectFromModel(rf, threshold='mean', prefit=True)
X_important = feat_sel.transform(X_test)

# 5. RFE with cross-validation
rfecv = RFECV(estimator=rf, step=1, cv=5, scoring='f1')
rfecv.fit(X_train, y_train)
print(f"Optimal features: {rfecv.n_features_}")

KEY TAKEAWAYS

  • Variance threshold first — eliminates near-constant features instantly and cheaply.
  • Mutual information detects non-linear relationships that Pearson correlation misses — use it for complex data.
  • Tree feature importance is fast and model-native but can be biased toward high-cardinality features — use SHAP for deeper analysis.
  • Feature selection is part of the pipeline — fit selectors on training data only, apply to test.
  • The curse of dimensionality: KNN, SVM, and K-Means degrade badly with many irrelevant features. Feature selection is not optional for these algorithms.

Dimensionality Reduction

3.5 hours

Why this matters: When you have hundreds or thousands of features, models overfit, training slows, and visualisation becomes impossible. Dimensionality reduction finds a lower-dimensional representation that preserves the most important structure. PCA is the workhorse for preprocessing; t-SNE and UMAP are the go-to tools for visualising high-dimensional data.

📓 Day6_Dimensionality_Reduction.ipynb

📐 PCA — Principal Component Analysis

Finds the directions of maximum variance in the data (principal components — PCs) and projects data onto them. PCs are orthogonal (uncorrelated) and ordered by how much variance they explain. PCA(n_components=0.95) keeps enough components to explain 95% of variance — automatically determines k. Always standardise before PCA (it's variance-based; different feature scales will dominate). Use for: preprocessing before linear models, removing collinearity, speeding up training on high-dimensional data.

📊 Understanding Explained Variance

pca.explained_variance_ratio_ tells you what fraction of total variance each PC captures. Plot cumulative explained variance to pick the "elbow" — the k where adding more PCs gives diminishing returns. Common targets: 95% for preprocessing, 2D or 3D for visualisation. The scree plot (variance vs. component number) helps visually identify the cutoff.

🎨 t-SNE — Visualising Clusters

t-Distributed Stochastic Neighbour Embedding preserves local structure (nearby points stay nearby) at the cost of global structure. Not for preprocessing — only for visualisation. Non-deterministic; run multiple times to confirm patterns. Key hyperparameter: perplexity (5–50) controls the neighbourhood size. Slow on large datasets — subsample to <50k points. Never use t-SNE components as model features.

🗺️ UMAP — Modern Alternative to t-SNE

Uniform Manifold Approximation and Projection is faster than t-SNE, better preserves global structure, and can be used for preprocessing (unlike t-SNE). umap.UMAP(n_components=2, n_neighbors=15, min_dist=0.1). Key params: n_neighbors (local vs global balance) and min_dist (cluster tightness). Works well as a preprocessing step for downstream clustering or classification.

🔢 Truncated SVD & Sparse Data

TruncatedSVD is PCA's cousin that works directly on sparse matrices (no centring required). Critical for text data (TF-IDF matrices are sparse). Also known as LSA (Latent Semantic Analysis) in NLP contexts. TruncatedSVD(n_components=100) reduces a 10,000-word TF-IDF matrix to 100 semantic dimensions. Used heavily in document similarity and recommendation systems.

from sklearn.preprocessing import StandardScaler
from sklearn.decomposition import PCA, TruncatedSVD
import matplotlib.pyplot as plt
import numpy as np

# PCA — always scale first
scaler = StandardScaler()
X_scaled = scaler.fit_transform(X_train)

# Find k to explain 95% variance
pca_full = PCA()
pca_full.fit(X_scaled)
cumvar = np.cumsum(pca_full.explained_variance_ratio_)
k = np.argmax(cumvar >= 0.95) + 1
print(f"Components for 95% variance: {k}")

# Scree plot
plt.figure(figsize=(10,4))
plt.plot(range(1, len(cumvar)+1), cumvar * 100)
plt.axhline(y=95, color='orange', linestyle='--', label='95% threshold')
plt.xlabel('Number of components')
plt.ylabel('Cumulative explained variance (%)')
plt.title('PCA Scree Plot')
plt.legend()
plt.tight_layout()
plt.show()

# Apply PCA with chosen k
pca = PCA(n_components=k)
X_pca = pca.fit_transform(X_scaled)

# t-SNE for visualisation only
from sklearn.manifold import TSNE
tsne = TSNE(n_components=2, perplexity=30, random_state=42, n_iter=1000)
X_2d = tsne.fit_transform(X_scaled[:5000])  # subsample

# UMAP (install: pip install umap-learn)
import umap
reducer = umap.UMAP(n_components=2, n_neighbors=15, min_dist=0.1, random_state=42)
X_umap = reducer.fit_transform(X_scaled)

DIMENSIONALITY REDUCTION SELECTION GUIDE

Preprocessing for linear models→PCA (removes collinearity)
Visualise clusters / classes→t-SNE or UMAP (2D/3D only)
Sparse data (text/TF-IDF)→TruncatedSVD
Preprocessing + speed + interpretable→PCA
Non-linear manifold, preserve local→t-SNE (vis only)
Non-linear + can use as features→UMAP

KEY TAKEAWAYS

  • Always StandardScale before PCA — PCA is variance-based, so unscaled features with large ranges will dominate the components.
  • t-SNE is only for visualisation — its components are not meaningful as model features. Never do PCA + t-SNE → model.
  • UMAP is faster than t-SNE, preserves global structure better, and can be used as a preprocessing step.
  • PCA components are interpretable via pca.components_ — each row shows how much each original feature contributes to a PC.
  • TruncatedSVD is the go-to for NLP text pipelines (TF-IDF matrices cannot be centred due to sparsity).

End-to-End Preprocessing Pipeline

4 hours

Why this matters: Applying transformations manually in the right order is error-prone and causes data leakage. Scikit-learn's Pipeline and ColumnTransformer bundle all preprocessing steps into a single object that can be fitted on training data and applied consistently to any new data — making deployment safe and reproducible.

📓 Day7_Preprocessing_Pipeline.ipynb

🔧 Pipeline — Sequential Step Chaining

Pipeline([('scaler', StandardScaler()), ('model', LogisticRegression())]) chains transformers and a final estimator. Calling pipeline.fit(X_train, y_train) fits all steps in sequence. pipeline.predict(X_test) transforms through all preprocessing steps then predicts. The key benefit: no data leakage — fit statistics come only from the data passed to that fit call. The entire pipeline can be pickled for deployment.

🗂️ ColumnTransformer — Different Treatment per Column Type

Numeric and categorical columns need different preprocessing. ColumnTransformer applies different transformers to different column subsets and concatenates results. remainder='passthrough' passes untouched columns through. remainder='drop' drops them. Use with make_column_selector(dtype_include='number') for automatic column type detection.

🔗 FunctionTransformer — Custom Logic in Pipeline

FunctionTransformer(np.log1p) wraps any function as a scikit-learn transformer. Use for custom feature transformations that don't have a dedicated sklearn class — domain-specific computations, interaction features, date component extraction. Pairs with Pipeline so custom logic participates in cross-validation correctly.

⚡ Pipeline + Cross-Validation = Safe Evaluation

Running cross_val_score(pipeline, X, y, cv=5) fits the entire pipeline inside each fold — preprocessing is re-fitted on training folds and applied to validation folds. This is the only safe way to evaluate preprocessing. If you preprocess outside cross-validation, your imputer and scaler see validation data statistics, causing optimistic evaluation scores.

🎯 GridSearchCV Over Pipeline Parameters

Use double underscore to address pipeline step parameters: {'columntransformer__num_pipeline__scaler': [StandardScaler(), RobustScaler()]}. GridSearchCV can search over which scaler to use, imputation strategy, number of PCA components, and model hyperparameters — all in one cross-validated sweep. The best pipeline is automatically refitted on all training data.

from sklearn.pipeline import Pipeline
from sklearn.compose import ColumnTransformer, make_column_selector
from sklearn.impute import SimpleImputer
from sklearn.preprocessing import StandardScaler, OneHotEncoder, OrdinalEncoder
from sklearn.decomposition import PCA
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import cross_val_score, GridSearchCV

# ── Define column groups
num_features = X_train.select_dtypes(include='number').columns.tolist()
cat_features = X_train.select_dtypes(include='object').columns.tolist()

# ── Numeric pipeline
num_pipe = Pipeline([
    ('imputer', SimpleImputer(strategy='median', add_indicator=True)),
    ('scaler',  StandardScaler()),
])

# ── Categorical pipeline
cat_pipe = Pipeline([
    ('imputer', SimpleImputer(strategy='most_frequent')),
    ('encoder', OneHotEncoder(drop='first', handle_unknown='ignore',
                              sparse_output=False)),
])

# ── Combine with ColumnTransformer
preprocessor = ColumnTransformer([
    ('num', num_pipe, num_features),
    ('cat', cat_pipe, cat_features),
], remainder='drop')

# ── Full pipeline: preprocessor → optional PCA → model
full_pipeline = Pipeline([
    ('preprocessor', preprocessor),
    ('pca', PCA(n_components=0.95)),          # optional step
    ('model', RandomForestClassifier(n_estimators=100, random_state=42)),
])

# ── Safe cross-validation (preprocessing re-fitted inside each fold)
cv_scores = cross_val_score(full_pipeline, X_train, y_train,
                             cv=5, scoring='f1_weighted')
print(f"CV F1: {cv_scores.mean():.3f} ± {cv_scores.std():.3f}")

# ── Hyperparameter search over pipeline steps
param_grid = {
    'preprocessor__num__imputer__strategy': ['mean', 'median'],
    'preprocessor__num__scaler': [StandardScaler(), RobustScaler()],
    'model__n_estimators': [100, 300],
    'model__max_depth': [None, 10, 20],
}
gs = GridSearchCV(full_pipeline, param_grid, cv=5,
                  scoring='f1_weighted', n_jobs=-1)
gs.fit(X_train, y_train)
print(f"Best params: {gs.best_params_}")

# ── Save pipeline for deployment
import joblib
joblib.dump(gs.best_estimator_, 'preprocessing_pipeline.pkl')

# ── Load and predict in one line
pipe = joblib.load('preprocessing_pipeline.pkl')
preds = pipe.predict(X_new)

Data Leakage Prevention Checklist

Fit imputers on training only → apply to test. Fit scalers on training only → apply to test. Fit encoders on training only → apply to test. Use cross_val_score(pipeline, ...) not cross_val_score(model, preprocessed_X, ...). Never apply target encoding without K-fold cross-validation. Never compute feature statistics from the full dataset before splitting.

KEY TAKEAWAYS

  • A Pipeline is the single most important tool in your ML engineering toolkit — it prevents leakage and makes deployment safe.
  • Always use cross_val_score(pipeline, ...) not cross_val_score(model, X_preprocessed, ...) — preprocessing must be inside the fold.
  • ColumnTransformer handles the fact that numeric and categorical columns need different treatment.
  • The fitted pipeline is deployable — save with joblib.dump(), load with joblib.load(). One file, all preprocessing.
  • GridSearchCV can search over preprocessing parameters too — not just model hyperparameters.

01 Missing Data What are MCAR, MAR, and MNAR, and why does the distinction matter? ▼
MCAR (Missing Completely At Random): missingness is independent of any value — safe to use simple imputation or drop rows. MAR (Missing At Random): probability of missing depends on other observed features — use model-based imputation like KNN or MICE. MNAR (Missing Not At Random): probability depends on the missing value itself — hardest case, requires domain knowledge or collecting the missing data. The distinction matters because using the wrong strategy introduces bias into your model.
02 Missing Data Why should you never fit an imputer on the full dataset before splitting? ▼
Fitting on the full dataset allows test-set statistics (e.g., the median) to influence the imputed values in the training set — this is data leakage. The model indirectly "sees" test data during training, producing overly optimistic evaluation metrics and poorer real-world performance. Always: split first, then imputer.fit_transform(X_train), then imputer.transform(X_test).
03 Missing Data When should you add a missing indicator column alongside imputation? ▼
Add a missing indicator when the fact that a value is missing is itself predictive of the target — i.e., when data is MAR or MNAR. For example, if high-income customers systematically skip the income field, the absence of income is informative. SimpleImputer(add_indicator=True) appends a binary column (feature_was_missing) before imputation. The model can then learn from both the imputed value and the missingness pattern.
04 Encoding Why does one-hot encoding require dropping one column (dummy variable trap)? ▼
One-hot encoding of K categories creates K columns that always sum to 1 — perfect linear dependency (multicollinearity). For linear models, this makes the feature matrix non-invertible and destabilises coefficient estimates. Dropping one column (drop='first') breaks the linear dependency. Tree-based models (Random Forest, XGBoost) don't invert matrices so they're not affected, but dropping is still good practice for memory and clarity.
05 Encoding What is target encoding and what is the risk? How do you mitigate it? ▼
Target encoding replaces each category with the mean of the target variable for that category (e.g., city → mean house price in that city). Very powerful for high-cardinality features. Risk: target leakage — if you compute target means on training data and then use those same samples to train the model, the model sees the target through the encoding. Mitigation: K-fold cross-encoding — for each training fold, compute the encoding from the out-of-fold rows. Use category_encoders.TargetEncoder or sklearn.preprocessing.TargetEncoder (sklearn ≥ 1.3) which handles this automatically.
06 Encoding When would you choose frequency encoding over one-hot encoding? ▼
Use frequency encoding when: (1) cardinality is high (100+ unique values) — OHE would create 99+ columns, causing dimensionality explosion. (2) You want to encode rare vs. common categories as a meaningful signal. (3) You need to avoid target leakage that target encoding introduces. Frequency encoding maps each category to its count/frequency in the training data: freq_map = X_train['col'].value_counts(); X['col_freq'] = X['col'].map(freq_map). Handle unseen categories with .fillna(0) or a small default value.
07 Scaling When would you choose RobustScaler over StandardScaler? ▼
Choose RobustScaler when the feature has significant outliers. StandardScaler uses mean and std — a single extreme value shifts the mean and inflates the std, compressing all "normal" values to a very narrow range. RobustScaler uses median and IQR (interquartile range), both of which are resistant to outliers. Typical use cases: income, transaction amounts, property prices, sensor readings with known noise spikes. If you're unsure, check df['col'].describe() — if the max is 5+ standard deviations from the mean, use RobustScaler.
08 Scaling Why don't tree-based models need feature scaling? ▼
Decision trees (and ensembles like Random Forest and XGBoost) split on threshold values: "is feature X > t?". The threshold is learned from data; it doesn't matter whether X is in [0,1] or [0,10000] — the relative ordering of values is preserved under any monotonic scaling. There is no distance calculation or gradient update that would be affected by scale. Linear models and KNN use distances/dot products that are scale-sensitive, so they require scaling. Neural networks are gradient-descent-based so they also require scaling for stable convergence.
09 Scaling What does PowerTransformer do and when should you apply it? ▼
PowerTransformer(method='yeo-johnson') applies a non-linear transformation that makes the distribution more Gaussian (symmetric, bell-curve-like). The Yeo-Johnson method works for any real values (including zero and negative). Box-Cox requires strictly positive values. Apply before StandardScaler when: (1) a numeric feature is highly skewed (e.g., income, prices, counts). (2) You're using a linear model — they assume Gaussian errors. Check skewness: df.skew() values > 1.0 suggest a power transform will help.
10 Feature Selection What is the curse of dimensionality and how does feature selection help? ▼
In high-dimensional spaces, all points become approximately equidistant from each other — distance loses meaning. To maintain the same data density, you need exponentially more samples as dimensions increase. Practical effects: KNN fails (can't find true neighbours), K-Means fails (centroids poorly defined), SVMs overfit. Feature selection removes irrelevant/redundant dimensions, keeping only the signal. This improves model generalisation, reduces training time, lowers memory usage, and makes models more interpretable.
11 Feature Selection What is the difference between filter, wrapper, and embedded feature selection methods? ▼
Filter methods evaluate features independently using statistics (variance, correlation, mutual information) — fast, model-agnostic, but ignore feature interactions. Examples: VarianceThreshold, SelectKBest. Wrapper methods use model performance to evaluate feature subsets — accurate but computationally expensive. Examples: RFE, RFECV. Embedded methods perform selection during model training — efficient and model-native. Examples: L1 regularisation (Lasso), tree feature importance, SelectFromModel. In practice: use filter methods first to remove obviously bad features, then embedded methods for the final selection.
12 Feature Selection Why might mutual information be better than Pearson correlation for feature selection? ▼
Pearson correlation measures linear relationships only. A feature can have zero correlation but a strong non-linear relationship with the target (e.g., quadratic: y = x²). Mutual information measures the total statistical dependency between two variables — it captures both linear and non-linear relationships. MI = 0 means truly independent; higher values mean stronger dependence. Use mutual_info_classif or mutual_info_regression via SelectKBest. Note: MI is more expensive to compute and requires enough data to estimate the joint distribution reliably.
13 PCA What are principal components and what does explained variance ratio mean? ▼
Principal components (PCs) are the directions of maximum variance in the data — orthogonal axes along which the data varies most. PC1 captures the most variance, PC2 the second-most (orthogonal to PC1), and so on. pca.explained_variance_ratio_[0] = the fraction of total variance captured by PC1. np.cumsum(pca.explained_variance_ratio_) gives cumulative variance — plot this (scree plot) to pick how many PCs to keep. Keeping 20 PCs out of 100 features might capture 95% of variance, discarding noise in the remaining 5%.
14 PCA Why must you standardise features before applying PCA? ▼
PCA finds directions of maximum variance. If one feature has range [0, 10000] and another [0, 1], the large-scale feature will dominate the first principal component purely due to its scale — not because it's more informative. StandardScaling ensures all features have unit variance before PCA, so the algorithm finds directions of maximum information across all features equally. Rule: always apply StandardScaler immediately before PCA in your pipeline.
15 PCA What is the key difference between t-SNE and UMAP, and when should you use each? ▼
t-SNE: preserves local structure (nearby points cluster), loses global structure (distances between clusters are meaningless). Non-deterministic, slow on large data, only for 2D/3D visualisation. Never use t-SNE components as model features. UMAP: faster, better preserves global structure, and can be used as a preprocessing step for downstream models. Both are great for exploring cluster structure visually. Use t-SNE when you want the best local cluster separation for a figure; use UMAP when you also want interpretable global layout or when you need to use the components downstream.
16 Pipeline What is the core purpose of scikit-learn's Pipeline and how does it prevent data leakage? ▼
Pipeline chains preprocessing steps and a model into a single object. When you call pipeline.fit(X_train, y_train), each step is fitted only on X_train and used to transform the input for the next step. When you call pipeline.predict(X_test), the same transformations (using training-data statistics) are applied to X_test — test data never influences the transformers. When used with cross_val_score(pipeline, X, y), the entire pipeline is re-fitted on each training fold — preventing test-fold statistics from leaking into training even during evaluation.
17 Pipeline What does ColumnTransformer do and why is it essential for real datasets? ▼
ColumnTransformer applies different transformations to different column subsets and concatenates the results into a single matrix. Essential because real datasets have mixed types: numeric columns need imputation + scaling; categorical columns need imputation + encoding; some columns might be dropped. Without it, you'd need to manually split, transform, and rejoin arrays — error-prone. remainder='passthrough' lets columns not mentioned pass through unchanged. make_column_selector(dtype_include='number') auto-detects column types.
18 Pipeline How do you tune preprocessing hyperparameters with GridSearchCV on a Pipeline? ▼
Use double underscore (__) to address nested pipeline parameters: step_name__parameter_name. For nested structures: columntransformer__num_pipeline__scaler. Example param grid: {'preprocessor__num__scaler': [StandardScaler(), RobustScaler()], 'model__max_depth': [5, 10, None]}. GridSearchCV will try every combination, re-fitting the full pipeline on each training fold. This lets you jointly optimise preprocessing strategy and model hyperparameters — the best combination might be RobustScaler + shallow tree rather than StandardScaler + deep tree.
19 Pipeline What is the complete data leakage prevention checklist for a preprocessing pipeline? ▼
1. Split first — always split before any preprocessing. 2. Fit transformers on train only — fit_transform(X_train), then transform(X_test). 3. Use Pipeline + cross_val_score — never evaluate on pre-processed data outside a fold. 4. Target encoding — use K-fold cross-encoding; never compute means on the full training set. 5. Missing indicators — fit on train, apply to test. 6. Feature selection — fit selectors on each training fold. 7. PCA/scaling — fit on train, apply to test. Rule of thumb: if the transformer has ever "seen" any label from the test set, you have leakage.
20 Pipeline How do you deploy a trained preprocessing pipeline to production? ▼
1. Save: joblib.dump(pipeline, 'model.pkl') — serialises all fitted transformers and the model as a single file. 2. Load: pipe = joblib.load('model.pkl') — restores the full pipeline with all fitted statistics (imputer medians, scaler means, encoder vocabularies, etc.). 3. Predict: pipe.predict(raw_df) — raw, unprocessed data goes in, predictions come out. All preprocessing happens automatically using training-time statistics. This is the gold standard for ML deployment — the pipeline handles everything from raw input to output in one call.
